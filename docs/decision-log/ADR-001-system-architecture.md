# ADR-001: 시스템 아키텍처

- 상태: Accepted
- 일자: 2026-06-25
- 관련: [ARCHITECTURE.md](../../ARCHITECTURE.md), [ADR-002](ADR-002-order-safety.md), [ADR-003](ADR-003-ai-trader-governance.md)

## 맥락

개인용 AI 자동매매 플랫폼을 설계한다. AI가 시장 데이터·지표·뉴스·전략 성과·포트폴리오를
분석해 투자 신호를 만들고, 검증을 거쳐 토스증권 Open API로 주문을 실행한다.
실 자금을 다루므로 **안전한 주문 실행, 책임 분리, 재현 가능한 검증**이 최우선이다.

## 결정

1. **Spring Boot 멀티모듈(Maven)** 구조를 채택하고, 신호 생성 / 위험 검증 / 주문 실행을
   **별도 모듈로 분리**한다.
   - `market-data-service`, `ai-trader-service`, `strategy-service`, `risk-engine`,
     `order-service`, `portfolio-service`, `audit-service`, `api-gateway`, `common`.
2. **저장소**: MySQL(영속), Redis(캐시·레이트리미트·락).
3. **전략 연구/백테스트**는 Python(research/)을 허용하되, 실거래 신호 생성은 백엔드 책임.
4. **프론트엔드는 최소 관리자 화면 또는 API 우선**으로 한다.
5. **신호와 주문 결과는 분리된 도메인/테이블**에 저장한다.

## 근거

- 모듈 분리는 "AI는 주문하지 않는다"는 불변식을 **컴파일/의존성 수준에서** 강제하기 쉽다.
- Python 연구 모듈 분리는 퀀트 연구 생산성과 재현성을 높인다.
- Redis는 중복/빈도 제한, 멱등 주문 처리에 적합하다.

## 대안과 기각 사유

- **단일 모놀리식 서비스**: 빠르지만 책임 경계가 흐려져 주문 안전성 보장이 약함 → 기각.
- **풀 마이크로서비스(분산 트랜잭션 중심)**: 개인 프로젝트에 운영 부담 과도 → 현 단계 기각,
  멀티모듈로 시작하고 필요 시 서비스 분리.

## 결과

- 명확한 책임 경계, 안전 규칙의 구조적 강제.
- 모듈 간 계약은 [API_CONTRACT.md](../../API_CONTRACT.md)로 관리.
- 트레이드오프: 초기 보일러플레이트 증가(모듈/계약 관리 비용).
