# ADR-002: 주문 안전성

- 상태: Accepted
- 일자: 2026-06-25
- 관련: [RISK_POLICY.md](../../RISK_POLICY.md), [SECURITY.md](../../SECURITY.md), [ADR-001](ADR-001-system-architecture.md)

## 맥락

실 자금 주문은 되돌리기 어렵고, 한 번의 결함이 직접적 자본 손실로 이어진다.
"수익보다 손실 제한"이라는 목표에 따라 주문 경로를 보수적으로 설계해야 한다.

## 결정

1. **단일 실행 지점**: 증권사 주문 API 호출은 `order-service`에서만 가능하다.
   - `ai-trader-service`는 빌드 의존성에 broker SDK/`order-service`를 **포함하지 않는다**.
2. **다층 검증 의무**: 모든 주문 후보는 `risk-engine`의 검증을 통과해야 한다.
   검증 항목과 거절 조건은 [RISK_POLICY.md](../../RISK_POLICY.md)에 정의한다.
3. **위험 우선**: 수익률이 높아도 한도를 넘는 주문은 실행하지 않는다.
4. **Kill Switch**: 전역/전략/종목 단위 즉시 차단 장치를 둔다. 자동 발동, 수동 해제.
5. **멱등성**: 주문은 멱등키로 중복 실행을 방지한다(Redis).
6. **모드 게이팅**: `BACKTEST → PAPER → SEMI_AUTO → AUTO(소액) → 확대` 순서만 허용.
7. **분리 저장**: 신호(`trading_signal`)와 주문/체결(`order`)을 분리 저장한다.
8. **비밀정보 격리**: 증권사 자격증명은 order-service에만 노출한다([SECURITY.md](../../SECURITY.md)).

## 근거

- 호출 지점을 하나로 모으면 감사·검증·차단을 한 곳에서 강제할 수 있다.
- 의존성 차단은 "AI가 실수로/악의로 주문"하는 클래스의 버그를 원천 제거한다.
- Kill Switch와 모드 게이팅은 사고 시 피해를 시간·금액으로 제한한다.

## 대안과 기각 사유

- **각 서비스가 직접 주문**: 검증 우회 위험, 감사 어려움 → 기각.
- **검증을 옵션(설정)으로**: 사고 시 비활성화 유혹 → 기각, 검증은 항상 필수.

## 결과

- 주문 경로가 단순·감사 가능해지고, 안전 규칙이 구조적으로 강제된다.
- 거절 케이스 테스트가 머지 차단 기준이 된다([CONTRIBUTING.md](../../CONTRIBUTING.md)).
- 트레이드오프: 자동화 유연성 일부 포기(의도된 보수성).
