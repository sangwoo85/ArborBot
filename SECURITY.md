# SECURITY.md — 보안 정책

본 문서는 **비밀정보 관리, 로그 마스킹, 감사 로그, 권한 분리, 운영 점검 항목**을 정의합니다.
금융 시스템 특성상 보안은 기능보다 우선합니다.

> **절대 규칙:** API Key, Secret, 계좌번호, Access Token은 **코드·로그·문서·Git에 절대 포함하지 않는다.**

---

## 1. 비밀정보(Secrets) 관리

### 1.1 대상

- 증권사 App Key / App Secret / Access Token / Refresh Token
- 계좌번호
- DB/Redis 자격증명
- AI 제공자 API Key
- 관리자 API 토큰

### 1.2 규칙

- 비밀정보는 **환경 변수** 또는 **Secret Manager**로만 주입한다. (`.env`는 로컬 전용, `.gitignore`로 제외)
- 저장소에는 `.env.example`만 두고, 값 자리에 `__SET_IN_LOCAL_ENV_ONLY__` 같은 플레이스홀더만 둔다.
- Access Token은 **메모리/짧은 수명 캐시**에만 두고, 평문 파일·DB·로그에 저장하지 않는다.
- 토큰 갱신(refresh)은 전용 컴포넌트에서 처리하고 만료 전 회전한다.
- 운영 환경에서는 Secret Manager(예: Vault/클라우드 KMS) 사용을 권장한다.
- 비밀정보가 실수로 커밋되면 **즉시 회전(폐기·재발급)** 하고 히스토리를 정리한다.

---

## 2. 로그 마스킹

- 로그에는 **계좌번호·토큰·키·개인 식별정보**를 절대 남기지 않는다.
- 공통 마스킹 유틸(common 모듈)을 통해 출력한다. (`LOG_MASKING_ENABLED=true`)
- 마스킹 규칙(예시):
  - 계좌번호 → `****-**-1234` (뒤 4자리만)
  - 토큰/키 → 전체 마스킹 또는 해시 앞 6자
  - 주문 식별자 → 필요한 만큼만
- 외부로 나가는 에러 응답에 내부 스택/비밀정보를 포함하지 않는다.
- 신호 입력 요약(`signal_input_snapshot`)에도 비밀/개인정보를 포함하지 않는다.

---

## 3. 감사 로그(Audit Log)

다음 이벤트는 **변경 불가(append-only)** 감사 로그로 기록한다.

| 카테고리 | 기록 항목 |
| --- | --- |
| 주문 | 주문 후보 생성, 위험 검증 결과(통과/거절+사유), 실행 결과 |
| 위험 한도 | 한도값 변경(이전/이후, 변경자, 사유) |
| Kill Switch | 발동/해제(주체, 시점, 사유) |
| 모드 변경 | BACKTEST/PAPER/SEMI_AUTO/AUTO 전환 |
| 전략 | 등록/승격/강등/중단 |
| 인증/권한 | 관리자 로그인, 권한 변경, 실패한 접근 |

- 감사 로그는 애플리케이션 일반 로그와 **분리 저장**한다.
- 시각은 신뢰 가능한 단일 시계(UTC 기준 저장, 표시 시 KST)로 기록한다.

---

## 4. 권한 분리

- **최소 권한 원칙**: 각 서비스/계정은 필요한 권한만 가진다.
- **주문 실행 권한**은 order-service에만 부여한다. ai-trader-service는 주문 권한이 없다.
- 증권사 자격증명은 order-service(또는 전용 broker 어댑터)에만 노출한다.
- 관리자 API(모드 변경, Kill Switch, 한도 변경)는 별도 인증·권한으로 보호한다.
- DB 계정은 서비스별 최소 권한(읽기/쓰기 분리 가능 시 분리).
- 위험한 조작(한도 상향, AUTO 승격, Kill Switch 해제)은 **사람의 명시적 승인**을 요구한다.

---

## 5. 네트워크 / 전송 보안

- 외부 API 통신은 TLS만 사용한다.
- 증권사/AI 제공자 호출은 타임아웃·재시도·서킷브레이커로 보호한다(무한 재시도 금지).
- 관리자 화면/API는 신뢰 네트워크 또는 인증 뒤에 둔다.

---

## 6. 의존성 / 코드 보안

- 의존성 취약점 스캔을 CI에 포함한다(예: OWASP dependency-check / `mvn` 플러그인).
- 비밀정보 누출 스캔(예: gitleaks)을 pre-commit/CI에 둔다.
- 외부 입력(증권사 응답, 뉴스 데이터)은 신뢰하지 않고 검증한다.

---

## 7. 운영 점검 체크리스트

**배포 전**
- [ ] `.env`/비밀정보가 커밋되지 않았는가 (gitleaks 통과)
- [ ] 로그 마스킹 활성화 및 샘플 로그에 비밀정보 없음 확인
- [ ] 기동 모드가 의도한 값인가 (`TRADING_MODE`)
- [ ] 위험 한도값이 정책과 일치하는가 ([RISK_POLICY.md](RISK_POLICY.md))
- [ ] Kill Switch 동작 테스트 통과

**운영 중(정기)**
- [ ] 토큰 만료/회전 정상
- [ ] 감사 로그 적재 정상, 변조 흔적 없음
- [ ] 일일 주문/손실 한도 대비 사용량 모니터링
- [ ] 비정상 주문/실패율 알람 동작
- [ ] 권한·접근 로그 검토

**사고 대응**
- [ ] Kill Switch 즉시 ON 절차 숙지
- [ ] 비밀정보 유출 시 즉시 회전 절차
- [ ] 이상 거래 발견 시 모드 강등(AUTO→SEMI_AUTO/PAPER) 절차

---

## 8. 취약점 신고

보안 취약점은 공개 이슈로 올리지 말고 비공개 채널(저장소 관리자)로 보고하십시오.
재현 절차와 영향 범위를 포함하되, **실제 비밀정보·계좌정보는 포함하지 마십시오.**
