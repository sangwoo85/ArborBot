# RISK_POLICY.md — 위험 관리 정책

본 문서는 **주문 차단 조건, 투자 비중 제한, 손실 한도, Kill Switch 정책**을 정의합니다.
Risk Engine과 Order Policy는 본 문서를 구현 기준으로 삼습니다.

> **최우선 원칙: 수익률이 높더라도 위험 한도를 초과하는 주문은 실행하지 않는다.**
> 개별 종목 수익보다 포트폴리오 전체 위험을 우선한다.

---

## 1. 검증 파이프라인

모든 주문 후보(OrderCandidate)는 아래 순서로 검증됩니다. **하나라도 실패하면 즉시 거절**하고 사유를 기록합니다.

```
0. Kill Switch 상태 확인        → ON 이면 무조건 거절
1. 거래 가능 시간 확인
2. 종목 상태 확인 (정지/관리/급등락/유동성)
3. 잔고 / 주문 가능 금액 확인
4. 1회 최대 주문 금액
5. 종목별 최대 투자 비중
6. 섹터별 최대 투자 비중
7. 일일 최대 주문 금액
8. 일일 최대 손실 한도
9. 최대 허용 손실률
10. 중복 주문 여부
11. 주문 빈도 제한
12. 신호 유효성(만료/신뢰도) 확인
→ 전부 통과 시에만 ALLOW
```

검증 결과는 `risk_evaluation`에 `ALLOW` 또는 `REJECT(위반 항목 목록)`로 저장됩니다.

## 2. 차단(거절) 조건 상세

| # | 조건 | 거절 기준(예시 기본값) | 비고 |
| --- | --- | --- | --- |
| 1 | **종목별 최대 투자 비중** | 단일 종목 평가금액 > 포트폴리오의 `10%` | `RISK_MAX_POSITION_PCT` |
| 2 | **섹터별 최대 투자 비중** | 단일 섹터 합계 > 포트폴리오의 `30%` | `RISK_MAX_SECTOR_PCT` |
| 3 | **1회 최대 주문 금액** | 단건 주문 > `1,000,000원` | `RISK_MAX_ORDER_AMOUNT` |
| 4 | **일일 최대 주문 금액** | 당일 누적 주문 > `5,000,000원` | `RISK_MAX_DAILY_ORDER_AMOUNT` |
| 5 | **일일 최대 손실 한도** | 당일 실현+평가 손실 ≥ `300,000원` | 도달 시 신규 매수 전면 차단 |
| 6 | **최대 허용 손실률** | 포지션 손실률 ≥ `5%` 또는 계좌 `MDD` 초과 | `RISK_MAX_LOSS_PCT` |
| 7 | **중복 주문** | 동일 종목·방향·미체결 주문 존재 | Redis 멱등키로 판정 |
| 8 | **주문 빈도 제한** | 분당 주문 > `5건` | `RISK_ORDER_RATE_PER_MIN` |
| 9 | **잔고/주문가능금액 부족** | 주문금액 > 주문가능금액 | 증권사 잔고 기준 |
| 10 | **거래 가능 시간 외** | 정규장/허용 시간 외 | 휴장일·동시호가 정책 포함 |
| 11 | **거래정지/관리종목** | 해당 지정 종목 | 매수 차단(청산 정책은 별도) |
| 12 | **급등락/유동성 부족** | 가격제한폭 근접·호가잔량 미달 | 슬리피지 방지 |
| 13 | **신호 만료/저신뢰** | `validUntil` 경과 또는 `confidence < 0.6` | [AI_TRADER_POLICY](AI_TRADER_POLICY.md) |
| 14 | **Kill Switch ON** | 전역/부분 스위치 활성 | 아래 4장 |

> 위 수치는 **기본값 예시**이며, 실제 값은 `.env`(`RISK_*`)와 운영 설정으로 관리합니다.
> 값 변경은 감사 로그에 기록하고, 상향 변경은 별도 승인이 필요합니다.

## 3. 투자 비중 / 한도 계산 규칙

- **포트폴리오 기준 금액** = (보유 평가금액 + 주문가능 현금). 한도는 이 기준 금액 대비 비율로 계산.
- 비중 검증은 **주문 체결 후 예상 비중**(post-trade)으로 평가한다. 즉, 이 주문이 체결되면 한도를 넘는지 본다.
- 매수 신호는 한도를 **엄격히** 적용하고, 위험 축소(매도/청산) 주문은 한도 검증을 완화할 수 있다(리스크 감소 방향).
- 손실 한도는 **실현손익 + 평가손익**을 합산해 일일 기준으로 집계한다.

## 4. Kill Switch 정책

Kill Switch는 시스템 전체 또는 일부를 즉시 정지시키는 안전장치입니다.

### 4.1 종류

| 종류 | 범위 | 트리거 |
| --- | --- | --- |
| **Global Kill Switch** | 모든 신규 주문 차단 | 수동(`KILL_SWITCH_ENABLED=true`) 또는 자동 |
| **Strategy Kill Switch** | 특정 전략의 주문 차단 | 전략 성과 악화 / 이상 신호 급증 |
| **Symbol Kill Switch** | 특정 종목 주문 차단 | 종목 이벤트/정지 |

### 4.2 자동 발동 조건(예시)

- 일일 손실 한도 도달
- 단위 시간 내 주문 실패율 급증(증권사 API 오류 폭증)
- 비정상 주문 패턴(중복/폭주) 탐지
- 시세/데이터 피드 중단으로 위험 평가 불가
- 계좌 잔고/포지션 조회 실패(상태 불명확)

### 4.3 동작 규칙

- Kill Switch가 ON이면 **신규 매수·신규 진입 주문은 무조건 거절**한다.
- 위험 축소(보유 포지션 청산/손절) 주문 허용 여부는 **명시적 정책 플래그**로 통제한다(기본: 수동 승인).
- 발동/해제는 **감사 로그**에 기록하며, 해제는 사람의 명시적 조치로만 가능하다(자동 해제 금지).
- 상태는 `kill_switch_state`에 저장하고 모든 서비스가 주문 전 조회한다.

## 5. 모드별 위험 적용

| 모드 | 위험 검증 | 비고 |
| --- | --- | --- |
| BACKTEST | 검증 로직 동일 적용(시뮬레이션) | 실주문 없음 |
| PAPER | 전체 검증 적용 + 가상 체결 | 실주문 없음 |
| SEMI_AUTO | 전체 검증 + **사람 최종 승인** | 승인 없이는 미실행 |
| AUTO | 전체 검증 후 자동 실행 | **한도를 최소값으로 시작** |

---

## 6. 테스트 요구사항

- 각 거절 조건마다 **거절 단위 테스트**가 존재해야 한다(통과/경계/초과 케이스).
- Kill Switch ON 상태에서 매수 주문이 거절되는 통합 테스트.
- 일일 손실 한도 도달 후 신규 매수 차단 시나리오 테스트.
- 중복/빈도 제한은 Redis 기반으로 동시성 테스트.

> 거절 로직의 누락은 곧 자본 손실 위험이므로, Risk Engine은 커버리지·리뷰 우선순위를 최고로 둡니다.
